# Documentation System Case Study

This site includes a fictional corporate documentation portal. The product names are not the main point. The purpose is to demonstrate how I can design a mature documentation system that feels similar to a company knowledge base.

## Project goal

Build a professional documentation website that shows:

- clear navigation for different user needs
- product documentation structure
- onboarding and quickstart content
- tutorial, how-to, explanation, and reference pages
- support and troubleshooting flows
- API documentation sections
- localization readiness
- GitHub Pages deployment

## Information architecture

The documentation lab is organized around user intent:

| Area | User question | Content type |
| --- | --- | --- |
| Getting Started | How do I begin? | onboarding, prerequisites, quickstart |
| Products | What does this product support? | overview, model pages, datasheets, manuals |
| Learn | How do I perform or understand tasks? | tutorials, how-to guides, explanation, reference |
| Support | How do I solve problems? | FAQ, troubleshooting, contact |
| API | How do developers integrate? | authentication, endpoints, errors |

## Diataxis mapping

The learning section can be mapped to Diataxis:

- **Tutorials** help users learn through guided practice.
- **How-to guides** help users complete a specific task.
- **Explanation** helps users understand concepts and decisions.
- **Reference** gives stable factual information.

This separation is important because users visit documentation with different levels of experience and different goals.

## What this demonstrates

This case study demonstrates my ability to:

- create a scalable documentation structure
- separate learning content from support content
- write navigation labels that match user intent
- build a docs-as-code site with Docusaurus
- think beyond individual pages and design a documentation system
