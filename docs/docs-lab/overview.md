# Corporate Documentation Lab

This section preserves the fictional corporate knowledge base I built as a documentation design exercise.

The product names are placeholders. The value is in the structure:

- onboarding
- product documentation
- support content
- tutorials
- how-to guides
- explanations
- reference pages
- API documentation
- localization readiness

## Why include fictional product docs?

A technical writing portfolio needs evidence. If I do not have permission to publish real company documents, I can still create realistic documentation samples that show my structure, writing judgment, and tooling ability.

## What hiring teams can evaluate

Hiring teams can look at this section to evaluate:

- whether the navigation is understandable
- whether content types are separated clearly
- whether product docs have consistent patterns
- whether the site feels like a maintainable knowledge base
- whether the API and support areas are discoverable
